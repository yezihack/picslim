# 智能图片压缩器详细开发计划（基于 PRD + Demo）

## 1. 文档信息

- 版本：v1.0
- 对应文档：`PRDs/SOFT.md`、`PRDs/TDD.md`
- 参考 Demo：`demo/index.html`、`demo/styles.css`、`demo/app.js`
- 目标：将当前 Demo 交互与视觉落地为可交付 MVP

## 2. 开发原则

1. **先骨架后逻辑**：先建立可运行框架与接口壳，再逐步接入真实处理逻辑
2. **先后端能力后前端展示**：先稳定任务模型和事件流，再替换前端模拟数据
3. **小步交付**：每个阶段都产出可验证结果
4. **可回归测试**：每个阶段定义验收口径，避免“做完不可测”

## 3. 里程碑与阶段目标

## Phase 0：工程脚手架与规范（2-3 天）

### 目标

搭建 Wails + Vue3 + Element Plus 工程，建立目录、规范、基础构建流程。

### 任务清单

1. 初始化 Wails 工程并确认三端构建命令
2. 初始化前端框架（Vue3 + Element Plus）并落地基础布局
3. 建立后端模块目录（scanner/scheduler/compressor/retry/report/preview）
4. 定义 DTO 与类型映射（前后端统一命名）
5. 建立基础日志模块和错误码常量

### 交付物

1. 可启动桌面应用（空逻辑）
2. 基础页面骨架（对齐 Demo 布局）
3. 基础模块目录和空实现

### 验收标准

1. Windows 本地可启动开发模式
2. 前后端可完成一次绑定调用（hello binding）
3. CI 或本地命令可完成构建

## Phase 1：任务核心能力（4-6 天）

### 目标

实现输入扫描、任务队列、并发压缩、重试、进度快照。

### 任务清单（后端）

1. `scanner`：输入路径扫描、格式过滤、文件总数统计
2. `scheduler`：任务状态机、worker pool、任务控制（开始/暂停/继续/取消）
3. `compressor`：按格式压缩（jpg/jpeg/png/webp，同格式输出）
4. `retry`：自动重试 3 次（1s/2s/4s）
5. `events`：推送 `task:progress`/`task:state_changed`/`task:error`
6. `dto`：`TaskConfig`、`ProgressSnapshot`、`FileJob`

### 任务清单（前端）

1. 输入输出配置面板接入真实 `createTask`
2. 任务控制按钮接入 `start/pause/resume/cancel`
3. 进度区接入后端事件，替换 `app.js` 的模拟时间

### 交付物

1. 可执行真实压缩任务（无预览）
2. 进度区数据为真实事件驱动

### 验收标准

1. 100 张样本可完成任务，进度和状态准确
2. 单文件失败能自动重试 3 次并落入失败列表
3. 暂停/继续/取消可用且状态转换正确

## Phase 2：结果、统计与报告（2-3 天）

### 目标

完善结果页、文件明细、统计聚合和 CSV 导出。

### 任务清单（后端）

1. `report`：汇总 bytesBefore/bytesAfter/savedPercent
2. 文件级结果存储与分页读取
3. CSV 报告导出
4. 打开输出目录能力

### 任务清单（前端）

1. 统计卡片接入真实结果数据
2. 文件明细表接入分页数据
3. 失败项展示与筛选
4. “导出 CSV 报告”“打开输出目录”按钮接入真实接口

### 交付物

1. 可用的结果页面
2. CSV 报告导出能力

### 验收标准

1. 统计值与文件明细可对账
2. CSV 报告字段完整且可在 Excel 打开
3. 输出目录可一键打开

## Phase 3：预览与体验完善（3-4 天）

### 目标

实现前后预览对比与当前 Demo 的交互视觉落地。

### 任务清单（后端）

1. `preview`：按需生成缩略图缓存
2. `getPreviewPair(taskId, jobId)` 接口
3. 预览失败兜底信息

### 任务清单（前端）

1. 预览区接入真实前后图片
2. 上一张/下一张联动文件明细
3. 继续保留“按钮主次层级”“吸底操作条”“动态时间展示”
4. 完善空态/异常态（暂无预览、加载中、读取失败）

### 交付物

1. 可交互预览区
2. 体验细节完整的 MVP UI

### 验收标准

1. 预览切换流畅，无明显卡顿
2. 大图/异常图有明确兜底提示
3. Demo 视觉元素在真实页面中完整保留

## Phase 4：稳定性、打包与发布（2-4 天）

### 目标

完成性能优化、回归测试、三端打包发布准备。

### 任务清单

1. 性能调优（并发数、I/O、事件频率）
2. 错误路径回归（权限不足、文件占用、输出冲突）
3. 测试补全（单测/集测/手测）
4. 三端构建验证（Windows/macOS/Linux）
5. 发布包说明文档（安装与使用）

### 交付物

1. 可分发安装包（至少 Windows 首发）
2. 回归测试记录和已知问题清单

### 验收标准

1. 关键流程通过率 >= 95%
2. 已知问题有明确影响范围和规避方案
3. 发布说明完整

## 4. 模块级任务拆分（可用于工单系统）

## 4.1 后端任务

1. **BE-01 scanner**
   - 输入：inputPaths
   - 输出：支持格式文件列表 + 过滤结果
   - 验收：扫描结果准确，过滤原因可追踪
   - 依赖：无
   - 预估：1 天

2. **BE-02 scheduler**
   - 输入：TaskConfig + 文件队列
   - 输出：任务状态与进度快照
   - 验收：状态机转换正确
   - 依赖：BE-01
   - 预估：1.5 天

3. **BE-03 compressor**
   - 输入：FileJob + preset
   - 输出：压缩后文件与文件结果
   - 验收：格式支持完整、收益守护生效
   - 依赖：BE-02
   - 预估：1.5 天

4. **BE-04 retry**
   - 输入：失败结果
   - 输出：重试后最终结果
   - 验收：最多 3 次，退避正确
   - 依赖：BE-03
   - 预估：0.5 天

5. **BE-05 report**
   - 输入：任务完成结果集
   - 输出：统计摘要 + CSV
   - 验收：统计准确、导出成功
   - 依赖：BE-03, BE-04
   - 预估：1 天

6. **BE-06 preview**
   - 输入：成功文件路径
   - 输出：预览对接口响应
   - 验收：可按需加载，失败可兜底
   - 依赖：BE-03
   - 预估：1 天

## 4.2 前端任务

1. **FE-01 页面框架迁移**
   - 输入：Demo 页面结构
   - 输出：Vue 组件化页面
   - 验收：布局一致、样式一致
   - 依赖：Phase 0
   - 预估：1 天

2. **FE-02 配置与控制**
   - 输入：绑定接口
   - 输出：输入输出选择、开始/暂停/继续/取消
   - 验收：按钮行为正确
   - 依赖：BE-02
   - 预估：1 天

3. **FE-03 进度与时间**
   - 输入：progress 事件
   - 输出：总进度、当前进度、已运行/剩余、ETA
   - 验收：秒级刷新，显示准确
   - 依赖：BE-02
   - 预估：1 天

4. **FE-04 统计与明细**
   - 输入：summary + file list
   - 输出：统计卡、明细表、失败状态
   - 验收：总计与明细可对账
   - 依赖：BE-05
   - 预估：1 天

5. **FE-05 预览区联动**
   - 输入：preview 接口 + file list
   - 输出：前后图预览、上一张/下一张
   - 验收：切换无错位，兜底可见
   - 依赖：BE-06
   - 预估：1 天

## 5. 联调顺序与替换策略

1. 第一步：连通 `createTask/getTaskSnapshot/startTask`（无事件）
2. 第二步：接入 `task:progress` 事件，替换 `app.js` 模拟进度
3. 第三步：接入结果统计与明细列表
4. 第四步：接入预览与导出按钮能力
5. 第五步：全流程压测与异常回归

## 6. 测试与验收计划

## 6.1 单元测试（Go）

1. 状态机流转
2. 重试策略
3. 统计计算
4. 路径与重名处理

## 6.2 集成测试

1. 混合格式批量输入
2. 自动输出目录与手动输出目录
3. 暂停/继续/取消
4. 自动重试 3 次
5. CSV 导出内容

## 6.3 UI 验收

1. 视觉与 Demo 对齐（毛玻璃、操作条、主次按钮）
2. 任务进度实时更新（含已运行/剩余）
3. 预览区操作可用
4. 错误提示清晰

## 7. 风险与应对

1. **跨平台图像库差异**
   - 应对：准备标准样本，三端回归同一批数据
2. **大目录性能波动**
   - 应对：流式扫描 + 并发上限 + 事件节流
3. **权限与文件占用问题**
   - 应对：创建阶段预检查 + 错误码细分 + 自动重试
4. **UI 事件风暴导致卡顿**
   - 应对：进度事件限频，前端按帧刷新

## 10. 依赖落地任务清单（BE-00x）

> 本节对应 `PRDs/TDD.md` 的“技术补充”章节，目标是把依赖选型落到可执行工单。原则：优先成熟开源包、版本可锁定、可替换、可回滚。

1. **BE-001 依赖基线初始化（go.mod）**
   - 目标：建立后端依赖基线与版本锁定策略
   - 关键依赖：
     - `github.com/disintegration/imaging v1.6.2`
     - `github.com/gin-gonic/gin`（稳定版本）
     - `github.com/alitto/pond/v2`
     - `github.com/cenkalti/backoff/v4`
     - `go.uber.org/zap`
     - `github.com/spf13/viper`
   - 产出：依赖清单文档 + `go.mod/go.sum` 锁定
   - 验收：可在干净环境 `go mod download` 成功
   - 依赖：无
   - 预估：0.5 天

2. **BE-002 imaging 适配层封装**
   - 目标：将图片压缩/缩放能力封装为内部接口，隔离第三方库
   - 产出：
     - `internal/compressor/adapter_imaging.go`
     - `internal/preview/thumbnail_imaging.go`
   - 约束：
     - 保持原格式输出（MVP 不转 WebP）
     - 支持收益守护（压后更大标记 `no_gain`）
   - 验收：样本图压缩/缩略图生成通过
   - 依赖：BE-001
   - 预估：1 天

3. **BE-003 gin 本地 HTTP 面落地（可选开关）**
   - 目标：提供调试/观测用本地 HTTP 服务，不替代 Wails binding
   - 产出：
     - `internal/httpapi/server.go`
     - 路由：`/api/v1/system`、`/api/v1/tasks`
   - 约束：
     - 支持 graceful shutdown
     - 支持中间件（request id/log/recovery）
   - 验收：本地接口可返回任务快照与健康状态
   - 依赖：BE-001, BE-02
   - 预估：1 天

4. **BE-004 pond 队列与并发池接入**
   - 目标：以 `pond` 替代手写并发池，支持有界队列与上下文取消
   - 产出：
     - `internal/scheduler/pool_pond.go`
     - 并发配置：`max(1, cpu-1)` + `queueSize`
   - 验收：
     - 支持暂停/继续/取消
     - 并发与队列上限生效
   - 依赖：BE-001, BE-02
   - 预估：1 天

5. **BE-005 backoff 重试策略接入**
   - 目标：统一重试策略为可配置指数退避（默认 1s/2s/4s）
   - 产出：
     - `internal/retry/retry_backoff.go`
   - 验收：
     - 自动重试 3 次
     - 可重试/不可重试错误分流正确
   - 依赖：BE-001, BE-04
   - 预估：0.5 天

6. **BE-006 zap 结构化日志接入**
   - 目标：统一日志格式，便于定位任务与文件级问题
   - 产出：
     - `internal/log/logger.go`
     - 关键字段：`taskId/jobId/state/attempt/errorCode/durationMs`
   - 验收：关键流程日志完整且可检索
   - 依赖：BE-001
   - 预估：0.5 天

7. **BE-007 viper 配置中心接入**
   - 目标：将并发、队列、重试、日志级别等参数配置化
   - 产出：
     - `internal/config/config.go`
     - 配置文件模板（dev/prod）
   - 验收：修改配置无需改代码即可生效（重启后）
   - 依赖：BE-001
   - 预估：0.5 天

8. **BE-008 依赖回归与升级策略**
   - 目标：建立依赖升级前后回归基线与回滚流程
   - 产出：
     - 回归脚本或清单（格式、路径、权限、异常）
     - 升级/回滚操作文档
   - 验收：依赖升级后可完成最小回归并产出记录
   - 依赖：BE-002 ~ BE-007
   - 预估：0.5 天

### 10.1 与现有工单映射关系

1. BE-001/006/007 建议在 Phase 0 执行
2. BE-002/004/005 建议在 Phase 1 执行
3. BE-003 建议在 Phase 2 执行（调试能力更有价值）
4. BE-008 建议在 Phase 4 执行（发布前稳定性保障）

### 10.2 验收补充（依赖维度）

1. 依赖版本锁定清晰，关键包可追溯
2. 不出现“上层模块直接散用第三方库”现象（需经过适配层）
3. 依赖替换成本可控（核心逻辑不与具体库强耦合）
